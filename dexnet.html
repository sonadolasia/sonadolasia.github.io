Dex-Net User Interface

January - April 2018

In Spring 2018, working as an undergratuate researcher in the UC Berkeley Automation Lab (AUTOLab), I was given a unique project to design and develop a graphical user interface for the Dexterity Network (Dex-Net). I worked in collaboration with Prof. Ken Goldberg and graduate students; Jeff Mahler, Mike Danielczuk and Matt Matl.

PROJECT OVERVIEW

The AUTOLab was invited to present the Dex-Net project at Amazon's invite-only MARS conference in Palm Springs. For the live demo the lab wanted a display that would sit on top of the robot, demonstrating the thought process of the project in real time. 

At the time, I was a member of the lab, assisting on various research projects. Knowing a bit about my interest in design, I was approached to help design this display. 

One of the key challenges with this project was creating a simple design for the display that could be understood by a non-technical audience. 

WHAT IS DEX-NET?

The Dexterity Network (Dex-Net) is a research project including code, datasets, and algorithms for generating datasets of synthetic point clouds, robot parallel-jaw grasps and metrics of grasp robustness based on physics for thousands of 3D object models to train machine learning-based methods to plan robot grasps.

UNDERSTANDING DISPLAY REQUIRMENTS

The first step of this project was meeting with the graduate students working on the Dex-Net project and discussing the goals for the display. We came up with a list of information that should be displayed, and briefly discussed the actual implementation - discussing a few different languages and libraries that could be used to develop it. I was also given the dimensions of the screen I would be designing for. 

DIMENSIONS:

  • 1440px x 900px

REQUIRED ELEMENTS: 

  • Confidence percentage - The robot's percentage of confidence that it will perform a successful grasp
  • Depth camera live video - Displaying exactly what the robot is seeing
  • Picks per hour - Given its current speed, how many objects the robot could potentially pick up each hour
  • Weight of object bin - Show the weight of the object bin, which gets smaller as the robot is more successful 
  • Robot current thought process - Current stage of the algorithm (paused, sensing, planning, moving, error recovery)
  • AUTOLab logo
  • Dex-Net logo
  • UC Berkeley logo

IDEATION

After discussing the goals of the display at length, I felt slightly overwhelemed. There were so many different data points the lab wanted to include on the display, and finding a way to make it clean and not number heavy seemed challenging. I began by sketching out some different ways to present the different elements, and thinking about the composition of the display.

INITIAL DESIGNS

Discussing further with Prof. Goldberg, we came up with a few ideas of how the elements should be presented. These included having the confidence percentage be the most significant number, possibly having the object bin weight be shown in a scale bar, and having the robot's thought process in a linear traffic light bar (lighting up the current state).

After coming up with a few basic ideas, I began mocking up the design in Illustrator. 

FIRST DESIGNS:

Centered depth footage, confidence in a rectangular box  Centered depth footage, confidence in a circular box 
Left aligned depth footage, confidence in a circular box, with percentage filled in. Weight displayed in a bar graph.  Left aligned depth footage, confidence in a circular box. Weight displayed in a bar graph. 

FEEDBACK:

  • Colors: Match the colors to the official UC Berkeley colors. Try different colors for the green, so it still stands out but is more coehsive.
  • Depth image at center: Found that the depth image is the clearest way to demonstrate how the robot is thinking. Want this to be large and center so it is visible from a distance. 
  • Disconnect and reduce the size of the pause and error recovery steps: The robot would mainly be in either the "sensing," "planning," or "moving" stage. And would hopefully never reach the "error recovery" stage.
  • Add a scale bar for the confidence percentage. Make the screen more symmetrical and show the confidence percentage in a more visual way.  

ITERATION & FINAL DESIGN

I incorporated this feedback into my next iteration of designs. In this iteration I changed the color palatte, using the official UC Berkeley colors, which I found added a nice contrast. I added in the second bar graph for confidence and increased the size of the three key steps. 

FEEDBACK:

  • Change "PROBE" to "PUSH": Describes the same action, but in simpler terms. 
  • Change "MG" to "GRAMS": Had the wrong metrics.
  • Make the scale bar easier to read. Change the bar to be filled with a solid color, remove tick bars, and make border thicker.
  • Match scale bars. Make the scale bars on either sides the same size. 
  • Increase font size of numbers and bottom status bar names: Better readability, and make numbers stand out more. 
  • Remove % sign from confidence, add a decimal point. Percent sign takes up a lot of room and isn't super neccessary, would rather have a more precise confidence value. 
  • Replace yellow bottom line with blue. Easier to see.

REFLECTION

This project was one of my favorite things I've had the chance to work on. I was able to see the power of design, and how I could use my technical knowlodge to demonstrate the complexities of this algorithm in a more understandable way. Looking back one thing I would have liked to do, was a a bit of user testing. I feel like the design may still be a bit technical, and would be understood by a larger audience with some user feedback. However at the time of this project, due to the secrecy of this project and the fast-paced timeline, it was not possible to test with anyone not working in the lab.
